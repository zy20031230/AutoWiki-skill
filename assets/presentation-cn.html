四大设计原则

🧠
认知深度
不是浅层摘要，而是从作者 Introduction 中提取的 CRGP 因素。 具有对比式先前认知 / 认知更新结构的批判性分析。 内置反模式防止空泛填充。
🔗
时间知识图
每篇资料在该领域的时间线中定位。演化链、 跨领域来源、时间张力。主题自动生成 时间演化章节。
🏠
自维护 Wiki
三级自治：静默 → 通知 → 确认。 25+ 项 lint 检查捕获断链、层级违规和过期数据。 Wiki 在日常使用中自我修复。
🎯
智能分类
每次主题分配前执行 3 问适配检查。 里程碑层级支持合并（<5）和提升（≥5）。 杜绝分类错误。

设计决策

为什么这样设计？

为什么用 Markdown 而不是数据库？
LLM 原生处理文本。无需 ORM，无需迁移。个人规模下 grep 就够了。Obsidian 渲染得很漂亮。
为什么用 Claude Code 作编译器？
KB 层零 Python 写入代码。LLM 用内置工具直接操作 markdown。编译器无需改代码即可适配。
为什么不用 RAG？
维护良好的索引 + grep 在约百篇规模下优于向量搜索。无需嵌入管道。Karpathy 也同意这一点。
为什么是一个 skill 而不是一个 app？
SKILL.md 就是架构——390 行编码了质量标准、反模式和工作流规则。无服务器、无基础设施。
自愈 Wiki

Lint 修复，Reorganize 演化

Wiki 不仅存储知识——它在出错时自我修复， 在你重新思考分类时自我重构。

🔧 Lint — 自修复
25+ 项检查在每次操作中运行，不仅仅是被命令时。 当 Agent 读取任何页面时，它会检测并静默修复：
› 断裂的 [[wikilinks]]
› Feeds ↔ Cluster 不匹配
› 孤立页面 / 空壳主题
› 层级深度 > 3
› 过期的 raw_path 引用
› 时间链断裂
› 子主题 ≥5 未提升
› 主题 >8 未再分簇
结构性修复静默执行。语义变更升级至确认级。
